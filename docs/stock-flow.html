物料申请-出入库流程说明（基于当前代码实现）

覆盖：Web-Admin + UniApp；含主设备/辅料；含申请/出库/入库/退库/收货；含状态机与常见场景。
新流程：申请 → 审批 → 自助领料 → 仓库确认出库
兜底：无申请快速出库 / 扫码出库（套装）
入库：手动入库 / SN导入 / 辅料批量入库
退库：申请 → 仓库收货确认（支持部分收货）
生成时间：2026-01-17（以当时代码为准）

1. 范围与术语

核心：主设备按SN逐台、辅料按数量
概念 系统实现（关键点）
物料 设备类型 Equipment，分两类：main_device（主设备）、 auxiliary（辅料/辅材）。
主设备实例 EquipmentInstance（每台一条）。主设备出入库靠实例状态流转；入库必须提供 SN(serial_number)。
辅料库存 不建实例，按数量走；在 Inventory/StockTransactionItem 里体现。
库存汇总 Inventory（按“仓库+物料”聚合）：current_stock（现存）、 available_stock（可用）、allocated_stock（已分配/在外）。
申请单 MaterialRequest(REQ-...) + 明细 MaterialRequestItem。
领料单/待确认出库 IssueDraft(ISD-...)： 主设备SN清单 IssueDraftSerial + 辅料行 IssueDraftItem。
出入库单 StockTransaction（入库/出库/退库/调整等）+ 明细 StockTransactionItem。
退库单 本质也是一张 StockTransaction(transaction_type=return)，其状态在 approval_status 字段（待收货/部分收货/已收货/拒收/取消）。
扫码出库(套装)的领料记录 PickupRecord：主要用于“我的设备/领料记录”展示与退库入口。
重要：扣库存发生点
在“新流程”里：申请/审批/自助领料只形成单据与待确认清单，不扣库存；真正扣库存发生在“仓库确认出库”。 因此“已批准”不等于“必然能发得出来”，库存不足会在出库确认阶段拦截并支持部分确认。

2. 流程开关（系统配置）

Web：库存管理 → 流程设置
配置键（stock_flow_settings） 默认 影响
enable_material_request true 关闭后：物料申请/自助领料/待确认出库相关接口会直接拒绝（页面入口也应不可用）。
request_requires_approval true 关闭后：申请提交即“系统自动全量批准”，申请单直接进入 approved（不占用库存）。
issue_requires_warehouse_confirm true 关闭后：领料单提交时会直接出库（由提交人作为操作人），不再等待仓库确认（慎用）。
allow_quick_stock_out true 关闭后：扫码出库/快速出库接口将返回“快捷出库已关闭”。

4. 三条主线流程总览

建议先看新流程，再看兜底与入库
主线A：新流程（申请 → 审批 → 自助领料 → 仓库确认出库 →（可选）退库收货）
  1. 申请人（UniApp）创建申请单（draft）→ 提交（submitted）。
  2. 仓库侧审批：可全量批准（approved）或部分批准（partially_approved），也可驳回（rejected）。
  3. 申请人创建/进入领料草稿（IssueDraft:draft）：主设备逐台扫码SN；辅料填写计划数量。
  4. 提交领料单：默认变为 pending_confirm，等待仓库确认出库（可部分确认、多次确认）。
  5. 仓库确认出库后，库存扣减（现存/可用减少，已分配增加），主设备实例变为 issued。
  6. 可选：发起退库申请（变为待收货），仓库收货确认后回库（主设备实例回 in_stock）。
关键限制：
领料扫码时会校验：SN存在、在申请仓库、实例状态为 in_stock、且型号在申请单中、且不超过“可领上限”。 仓库确认出库时若库存不足会返回短缺清单，可改为“部分确认”。
主线B：无申请直接出库（兜底）
  • 快速出库（Web/UniApp）：仓库/管理员录入主设备SN列表 + 辅料数量 → 生成出库单并扣库存。
  • 扫码出库（套装出库，UniApp“我的设备”）：扫主设备条码/SN → 选套装 → 系统按套装明细一次性出库并产生领料记录（用于“我的设备/领料记录/退库入口”）。
建议
生产环境通常保留“快速出库/扫码出库”作为兜底，但尽量让现场人员走“申请→领料→仓库确认”的可追溯链路。
主线C：入库（手动入库 / SN批量导入 / 辅料批量入库）
  • 手动入库：主设备必须SN，且数量必须为1；辅料只填数量。
  • SN批量导入：导入成功会生成导入记录，并额外生成一条入库单据（便于在出入库记录中追溯）。
  • 辅料批量入库：前端解析Excel后直接调用“入库”接口创建入库单。

6. 状态机（按状态→可执行动作）

四张：申请单 / 领料单 / 退库单 / 主设备实例

6.1 申请单（MaterialRequest.status）

显示值为状态字段的真实取值（如 draft）。审批关闭时，“提交”可直接进入 approved。

6.2 领料单（IssueDraft.status）

“部分确认/多次确认”是领料单的核心能力，用于解决库存不足或分批发料场景。

6.3 退库单（return 的 approval_status）

退库单“收货确认”支持部分收货；一旦部分收货发生，就禁止取消/拒收整单。

6.4 主设备实例（EquipmentInstance.status）

仅展示与“出入库/检查/退库”强相关的状态；另外还有 damaged/repairing/scrapped 等终态/特殊态。

7. 角色动线（操作手册）

按“谁来做”说明
7.1 申请人（现场人员，UniApp为主）
  1. 创建申请：进入“物料申请 → 新建申请”，选择仓库，添加明细（主设备/辅料）。 可先保存草稿（draft），确认后提交（submitted）。
  2. 等待审批：申请详情页会提示“待审批”。仓库驳回会显示原因。
  3. 开始领料：当申请状态为 approved/partially_approved 且“剩余可领>0”时， 点“开始领料”自动创建领料草稿（IssueDraft:draft）。
  4. 领料草稿：
    • 主设备：逐台扫码/输入SN；系统会校验是否在申请仓库且在库（in_stock）。
    • 辅料：填写“本次计划领料数量”，不能超过“剩余可领上限”。
  5. 提交领料单：默认提交后变为 pending_confirm，等待仓库确认出库； 若系统配置关闭“仓库确认出库”，提交即出库成功。
  6. 退库（需要时）：
    • 从“我的设备”里点设备进入退库流程，或从“退库申请 → 新建退库”按出库单选择退库内容。
    • 若提示“需先解绑检查项”，可按提示执行“一键解绑”（会清空检查内容并删除相关照片）。
7.2 仓库管理员 / 管理员 / 项目经理（仓库侧，Web-Admin为主）
权限口径
后端鉴权将 manager 视作 admin 等同权限；仓库管理员通常仅能操作/查看自己管理的仓库。
  1. 入库：库存管理 → 入库管理。主设备必须SN且数量=1；辅料可直接填数量；批量导入会生成入库记录。
  2. 审批申请单：库存管理 → 物料申请。进入详情后对每行填“批准数量”并通过，或驳回并填写原因。
  3. 确认出库（关键）：库存管理 → 待确认出库 → 出库确认。勾选本次确认SN、填写辅料确认数量后确认出库。 若库存不足会给出短缺清单，可改为部分确认。
  4. 退库收货：库存管理 → 退库收货。支持部分收货；主设备按 SN 勾选收货（可扫码快速选择），辅料按数量收货。 可拒收（仅待收货且未发生部分收货）。
  5. 无申请出库兜底：库存管理 → 快速出库。录入主设备SN列表 + 辅料数量（可选指定领取人）完成出库。

8. 场景与异常处理

建议用“展开全部场景”快速浏览
场景1：库存不足（确认出库/快速出库/扫码出库）
触发点：仓库确认出库、快速出库、扫码出库。
表现：返回“库存不足”并附带短缺清单（需要/可用/现存）。
处理建议：改为部分确认、减少辅料数量、或拆单/等补货后再确认。
场景2：部分批准 / 部分确认出库（分批发料）
部分批准：仓库审批时按行填写批准数量（≤申请数量）。
部分确认：出库确认可仅勾选部分SN、仅确认部分辅料数量，多次确认直到发完。
系统会计算“待确认(pending)与剩余可发(remaining)”来限制再次领料/确认。
场景3：申请/领料的取消与驳回限制
  • 申请单仅 draft/submitted 可取消。
  • 领料单取消：仅 draft/pending_confirm 且“未发生任何部分确认”可取消。
  • 领料单驳回：仅 pending_confirm 且“未发生任何部分确认”可驳回整单。
场景4：主设备退库受“检查绑定”约束（需解绑/不可解绑）
预览可能返回两类提示：
  • need_unbind：存在可解绑的设备级检查项，可执行“一键解绑”（会清空检查内容并删除照片文件）。
  • unbind_blocked：存在不可解绑/不属于当前用户的绑定（如检查已提交/审核等），禁止发起退库。
目的：避免“设备在检查流程中”与“库存退库”并发造成账实错乱。
场景5：退库支持“只退辅料/只退主设备/部分退辅料”，且可多次退
  • 辅料：按数量退，且不允许超出“原出库数量 - 已申请退库数量”。
  • 主设备：逐台SN；同一台设备在 return_pending_receive 时不可再次发起退库。
  • 可多次发起退库单，直到可退数量归零。
场景6：退库收货支持部分收货，但一旦部分收货发生就不能拒收/取消整单
收货确认：主设备按 SN 勾选本次收货；辅料逐行输入“本次收货数量”。
主设备收货：支持扫码快速定位并勾选 SN（仅可勾选待收货的 SN）。
拒收：仅 pending_receive 且未发生部分收货才允许。
场景7：扫码出库（套装）与快速出库的差异
  • 扫码出库：基于“主设备 → 套装配置”生成出库明细，并生成领料记录（PickupRecord）。
  • 快速出库：主设备SN列表 + 辅料数量自定义，支持指定领取人（issued_to）。
  • 两者都属于“无申请出库”的兜底路径，受开关 allow_quick_stock_out 控制。
场景8：SN重复/导错入库怎么处理（撤销入库+重导）
SN批量导入/手动入库都会校验SN唯一性；重复会被拦截或跳过。
若导错且设备仍在库（in_stock 且未出库），仓库/管理员可执行“撤销入库（作废实例）”，系统会：
  • 将实例标记为作废（释放SN以便重导）
  • 生成一条“调整”单据记录这次撤销
  • 回扣对应库存数量
