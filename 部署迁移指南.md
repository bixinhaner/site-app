# 部署迁移指南

## 📌 适用场景

本指南适用于在新机器上部署应用，或从10月16日之前的版本升级到最新版本。

---

## ✅ 快速检查

**问题**: 我是否需要执行数据库迁移？

**检查方法**:
```bash
cd /path/to/site-app
sqlite3 site_manager.db "SELECT sql FROM sqlite_master WHERE name='user_logs';"
```

**结果判断**:
- 如果返回表结构 → **不需要迁移**（已经是最新版本）
- 如果返回空 → **需要执行迁移**

---

## 🚀 方案一：全新部署（推荐给新机器）

### 步骤

1. **克隆代码**
```bash
git clone <repository_url>
cd site-app
git pull origin main
```

2. **安装依赖**
```bash
# 后端
cd backend
python3 -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt

# 前端（如需）
cd ../web-admin
npm install

cd ../uniapp-site-manager
npm install
```

3. **自动创建数据库**
```bash
cd backend
python start_backend.py
```

SQLAlchemy会自动创建所有表结构（包括最新的变更）。

4. **初始化数据**（可选）
```bash
# 创建默认用户等
python init_db.py  # 如果有初始化脚本
```

---

## 🔄 方案二：升级现有部署（已有数据）

### 前置准备

1. **停止应用**
```bash
pkill -f uvicorn
pkill -f "npm run dev"
```

2. **备份数据库**
```bash
cd /path/to/site-app
cp site_manager.db site_manager.db.backup_$(date +%Y%m%d_%H%M%S)
```

3. **拉取最新代码**
```bash
git pull origin main
```

## ⚠️ 常见问题

### Q1: 迁移脚本报错 "duplicate column name"
**原因**: 字段已存在  
**解决**: 正常现象，可以忽略继续执行，或使用Python迁移脚本（自动跳过）

### Q2: 应用启动后报错 "no such column"
**原因**: 迁移未完全执行  
**解决**: 重新执行迁移脚本，检查所有字段是否添加成功

### Q3: 现有数据会丢失吗？
**答案**: 不会。迁移只添加新字段和新表，不删除或修改现有数据

### Q4: 可以回滚吗？
**答案**: 可以。恢复备份数据库：
```bash
cp site_manager.db.backup_YYYYMMDD_HHMMSS site_manager.db
```
然后回滚代码：
```bash
git checkout <旧版本commit>
```

### Q5: MySQL/PostgreSQL怎么办？
**答案**: 原理相同，但SQL语法略有不同。参考`数据库变更汇总-10月16日至今.md`中的MySQL脚本。

### Q6: 已经运行start_backend.py，为什么还缺少字段？
**原因**: SQLAlchemy不会自动添加字段到已存在的表  
**解决**: 必须手动执行迁移脚本
